### A. The Mediator Pattern (`ContextManager`)
- **Analysis**: The `ContextManager` acts as the "Heart" of the application, utilizing a strict Mediator pattern. Services (`DatabaseService`, `RAGService`, `ReportEngine`) do not communicate with each other directly; they only interact through the `ContextManager`.
- **Benefit**: This isolates state and ensures that forensic context (history, active databases) remains synchronized. Tool execution is safely dispatched via an explicit handler map (`self.tool_handlers`).

### B. The Investigative Pipeline (`QueryProcessor`)
- **Analysis**: The `QueryProcessor` operates as the "Central Nervous System", executing a strict 8-stage pipeline:
  1. Intent Interception
  2. Data Ingestion
  3. Forensic Keyword Analysis (via `IntentEngine`)
  4. Knowledge Base Lookup (via `RAGService`)
  5. Prompt Engineering
  6. AI Reasoning
  7. Verification via Forensic Tools
  8. Final Forensic Synthesis
- **Resilience Feature**: The implementation of `_run_python_triage` allows for a hardcoded, AI-independent execution path. This guarantees an initial comprehensive system report without relying on multi-turn LLM tool calls or API quotas.

## 2. Security Posture: The "Ghassan Elsman Boundary"

EYE enforces strict boundaries to protect digital evidence integrity.

### A. Data Integrity (Strict Read-Only Enforcement)
The engine's safety is rooted in its **unidirectional data flow** and multi-layered database protection in the `ForensicDatabaseService`:
1.  **Connection Level**: `PRAGMA query_only = ON` is enforced immediately upon connection.
2.  **Application Level**: A rigorous regex-based SQL keyword validator actively rejects modifying commands (`DROP`, `UPDATE`, `INSERT`, `ALTER`, etc.) before they reach the SQLite driver.
- **Verdict**: There is no path for the AI to "inject" malicious code into the system or tamper with parsed forensic artifacts.

### B. The Reporting Mandate
- **Analysis**: The `_build_synthesis_prompt` enforces the **'Ghassan Elsman Protocol'**. It explicitly instructs the AI to never summarize out crucial details (timestamps, PIDs, exact paths) and to always provide a chronological timeline. This prevents the AI from generating "fluff" and forces it to act as a technical forensic reporter.

### C. Credential Management
- **Security**: EYE utilizes OS-native keychains (Windows Credential Manager). API keys are never stored in plain text in the `eye_config.json`.

## 3. Model Backend Strategy (The Strategy Pattern)

The `ModelRouter` implements a clean Strategy pattern, categorizing AI communication into three distinct integration types:

1. **`cloud_api`**: Native SDKs talking to cloud services over HTTPS (OpenAI, Anthropic, Gemini).
2. **`local_server`**: REST API clients for local network AI services (Ollama, LM Studio).
3. **`local_cli`**: Subprocess wrappers for command-line AI tools.

### A. `local_cli` vs. `cloud_api`
| Factor | `local_cli` (e.g., Llama.cpp) | `cloud_api` (e.g., OpenAI) |
| :--- | :--- | :--- |
| **Privacy** | High (Data stays on disk*) | Low (Metadata leaves machine) |
| **Speed** | Variable (Subprocess overhead) | High (Fast inference) |
| **Control** | Full (User controls executable) | Low (Dependent on provider) |
| **Stability** | Medium (Shell environment deps) | High (Standardized HTTP) |

*\*Note: Even with local_cli, if the CLI agent itself calls a cloud API, data is still egressed.*

### B. The `GenericCLIBackend` Bottleneck
- **Analysis**: Spawning a shell process (`subprocess.run`) for every query introduces significant I/O and process overhead compared to maintaining a persistent socket or HTTP keep-alive (like the `local_server` approach).
- **Recommendation**: Transition `local_cli` to a "Long-Running Agent" mode where the process stays alive in the background, communicating via standard input/output streams continuously.

---

## 4. The Validation Layer: Performance vs. Integrity

### A. JSON Schema Overhead
- **Analysis**: EYE performs full schema validation on every configuration load. 
- **Impact**: While validation adds marginal latency (5-10ms), it prevents "Dead Ends" where the engine might start but crash mid-query due to a missing path.
- **Verdict**: A necessary trade-off that ensures forensic reliability over raw startup speed.

### B. AI Error Recovery Loop
- **Analysis**: The `QueryProcessor` implements a graceful degradation path (`_handle_generation_failure`). If an API quota is exhausted (e.g., HTTP 429), it catches the error and dynamically presents the user with UI Action Chips to instantly switch to an alternative local or cloud model.

---

## 5. Token Economy & State Management

- **RAG vs. History**: Currently, EYE allocates token budgets (e.g., `conversation_history`: 8000, `system_prompt`: 4000). 
- **Problem**: If the `rag_context` budget is too small, complex artifacts (like massive MFT records or Event Logs) are truncated before the AI can see the full timeline.
- **Security Implication**: Truncated data can lead to "Blind Spots" in the final report, where the AI misses a crucial event.
- **Mitigation strategy**: EYE relies on the TOON (Token-Oriented Object Notation) Engine to compress database query results, but dynamic compression based on remaining context window size is required to fully eliminate blind spots.
